# Ресторан - модульное CLI-приложение на Dart

Консольное приложение для управления рестораном с хранением данных в SQLite.

## Выбранная предметная область

**Ресторан** - система управления рестораном, включающая роли сотрудников, учёт пользователей (сотрудников), меню блюд и заказы.

### Сущности:
1. **Роли** (Role) - должности сотрудников (официант, повар, администратор и т.д.)
2. **Пользователи** (User) - сотрудники ресторана с привязкой к роли
3. **Меню** (MenuItem) - блюда и напитки с ценой и описанием
4. **Заказы** (Order) - заказы с привязкой к пользователю и пункту меню

### Связи между сущностями:
- Пользователь - Роль (many-to-one: каждый пользователь имеет одну роль)
- Заказ - Пользователь (many-to-one: каждый заказ оформляется одним пользователем)
- Заказ - Пункт меню (many-to-one: каждый заказ содержит один пункт меню с количеством)

## Что вынесено в каждый слой и почему

### Domain (lib/src/domain/)
Содержит только модели и валидаторы. Никакого SQL и stdin.
- **models/** - классы-сущности с полями, toMap(), fromMap(). Не зависят от БД и консоли.
- **validators/** - чистые функции валидации. Могут тестироваться без БД.

**Почему:** Если заменить SQLite на JSON или сменить консоль на веб, этот слой не меняется.

### Data (lib/src/data/)
Содержит работу с БД и репозитории. Импортирует domain, но не cli.
- **database.dart** - открытие/закрытие SQLite, создание таблиц с FOREIGN KEY.
- **repositories/** - классы с CRUD-операциями (create, getAll, getById, update, delete).

**Почему:** Если заменить SQLite на другой способ хранения, меняется только этот слой.

### CLI (lib/src/cli/)
Содержит меню и ввод/вывод. Импортирует и domain, и data.
- **menu.dart** - класс Menu с циклом while, switch-case, вызов репозиториев.
- **input_helper.dart** - статические методы askString, askPositiveInt, askPositiveDouble, askDateTime с повторным запросом при ошибке.

**Почему:** Если заменить консоль на веб-интерфейс, перепишется только этот слой.

## Реализованная функциональность

Для каждой из 4 сущностей реализован полный CRUD:
- **Роли:** список, добавить, обновить, удалить
- **Пользователи:** список, добавить, обновить, удалить (с проверкой существования роли)
- **Меню:** список, добавить, обновить, удалить
- **Заказы:** список, добавить, обновить, удалить (с проверкой существования пользователя и пункта меню)
- **Показать ВСЁ из БД** — вывод всех 4 таблиц одновременно

## Какие валидации реализованы

### 1. Обязательное текстовое поле (не пустое после trim())
- **TextValidator.validateNonEmptyTrimmed()** — проверяет, что строка не null, не пустая и не состоит только из пробелов.
- Используется для: название роли, ФИО, телефон, название блюда, описание.

### 2. Числовое поле > 0
- **NumberValidator.validatePositiveInt()** — проверяет, что строка парсится в int > 0.
- **NumberValidator.validatePositiveDouble()** — проверяет, что строка парсится в double > 0.
- Используется для: цена блюда, количество в заказе, ID.

### 3. Дата/время корректно парсится в DateTime
- **DateTimeValidator.validateDateTime()** — проверяет, что строка парсится в DateTime.
- Используется для: дата заказа.

## Перечень тестов

### domain_test.dart — Тесты моделей
- Role: toMap / fromMap корректно преобразуют данные
- User: toMap / fromMap корректно преобразуют данные
- MenuItem: toMap / fromMap корректно преобразуют данные
- Order: toMap / fromMap корректно преобразуют данные (включая DateTime)

### validation_test.dart — Тесты валидации
- TextValidator: пустая строка возвращает ошибку
- TextValidator: строка из пробелов возвращает ошибку
- TextValidator: непустая строка проходит валидацию
- NumberValidator: не-число возвращает ошибку
- NumberValidator: отрицательное число возвращает ошибку
- NumberValidator: ноль возвращает ошибку
- NumberValidator: положительное число проходит валидацию
- DateTimeValidator: некорректная дата возвращает ошибку
- DateTimeValidator: корректная дата проходит валидацию

### data_test.dart - Тесты БД
domain_test.dart - тест модели (toMap / fromMap)
В данном файле реализован тест, проверяющий корректность преобразования модели User. Создаётся объект User, затем вызывается метод toMap(), после чего на основе полученной Map создаётся новый объект через fromMap(). Проверяется, что значения полей id, fullName, phone и roleId совпадают с исходными. Тест подтверждает корректность работы методов преобразования модели

validation_test.dart - тест валидаторов
В этом файле проверяется работа валидаторов текстовых и числовых данных. Для TextValidator проверяется, что строка, состоящая только из пробелов, возвращает ошибку, а корректная непустая строка проходит валидацию. Для NumberValidator проверяется, что отрицательное число считается невалидным, а положительное число проходит проверку. Тест подтверждает корректность обработки валидных и невалидных входных данных

data_test.dart - тест работы базы данных
В данном тесте проверяется работа RoleRepository с использованием временной базы данных SQLite в памяти. Создаётся база данных через DatabaseHelper.inMemory(), затем через репозиторий добавляется роль. После этого роль считывается по идентификатору с помощью метода getById(). Проверяется, что объект успешно найден и его имя совпадает с ожидаемым значением. В конце соединение с базой данных закрывается. Тест подтверждает корректную работу операций вставки и чтения данных в слое data
